Module 06

Architecture, identity & data

How the console maps onto the four-cloud estate you walked us through — who instruments what, how identity works without PII, and how the data lands in BigQuery.

Wallet App CLOUD 1

Softspace · AWS · native iOS Swift / Android Kotlin
  • Native SDK · auto-captures hero carousel & Hot Deals
  • Instrumentation requires AEON Credit Service approval

Membership CLOUD 2

Hitachi · AWS · React WebView in native
  • JS tag in the WebView · covers AEON Points pages
  • CSP whitelist for the tag domain, coordinated with Hitachi
  • Instrumentation requires AEON Credit Service approval

Super Deals CLOUD 3

Eber · SaaS PWA rendered in the Hitachi WebView
  • Same JS tag, deployed in the Eber PWA
  • Vendor request handled directly by AEON360

Ad Portal CLOUD 4

AEON360 · GCP · Java backend
  • Server-side join: placement ID, name, layout, slot — campaign & advertiser stay out of the front end
  • AEON360-owned, no external approval needed

Pseudonymous identity — no email

AMP ID (hashed) is the master identity — no email collected. Sessions, devices and return visits stitch retroactively the moment a user logs in. If onboarding changes and a new ID is introduced later, the switch is non-destructive.

AMP ID master email not required user properties: tier, BU flags

Masked replay for support

Session replay is a masked reconstruction, not a recording — every sensitive element hidden by default. A support agent can ask a caller for their AMP ID and replay exactly what happened, without exposing personal data.

masked by default find session by AMP ID

BigQuery, natively

All behavioral data is queryable from BigQuery in place — no duplicated storage cost, you pay only BigQuery compute. Egress applies only if you copy data out. 13 months retention; Milk's team joins it to transactional data for modeling.

no duplicate storage 13-month retention
Rollout

Three phases · monetization live inside 3 months

No proof-of-concept detour. The approvals a pilot would need — AEON Credit sign-off, the Hitachi CSP whitelist — are the same approvals production needs. Secure them once, go live once: auto-capture validates the tracking on real traffic in week one, which is exactly what a POC would have spent a month proving.

PHASE 1 · weeks 1–4

Instrument & validate in production

  • Tag the membership page (web view — fastest path) + wallet home banner, once AEON Credit sign-off lands
  • Impressions, taps, CTR per banner & slot
  • Daily uniques, active periods, lingering time
  • Cross-surface journey Cloud 1 → 2 → 3
PHASE 2 · months 2–3

Switch on monetization

  • Conversion events: voucher redemption, points exchange, purchase
  • Push-notification attribution
  • Partner campaign reports with scoped access
  • Rate card backed by per-slot CTR data
PHASE 3 · beyond

Scale across the estate

  • All mini-programs as they launch — the playbook proven with super-apps like GCash
  • Cross-BU journeys & delayed conversions across AEON operating companies
  • BigQuery modeling: LTV, propensity, audience packages for partners
⚡

Why this page replaces the pilot

A 4-week POC consumes the same cross-company coordination as a production go-live — then asks everyone to do it twice. Capture is automatic and retroactive, so there is no tagging plan to de-risk, and the mini-program playbook is already proven at scale with super-apps like GCash. Week-one production data answers every item on the checklist at the top of this page, a quarter sooner.
